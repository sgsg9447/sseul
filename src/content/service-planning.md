## 1. 프로젝트 개요

> 기존 사이트는 정보와 기능을 갖췄지만, **콘텐츠가 데이터가 아닌 이미지·게시글로만 존재**하고 **사용자 행동 흐름과 정보 위계가 설계되지 않아**, 예비 수강생은 신청을 판단·실행하기 어렵고 운영자는 반복 수작업에 묶여 있다.

토이 프로젝트가 아니라 **실제 운영 사이트와 실제 이해관계자**를 대상으로 한 분석에서 출발했다. 실제 학원 운영자를 인터뷰해 콘텐츠 수정 방식·상담 유입·신청자 관리의 반복 업무를 파악했고, 예비 수강생 입장에서 메인→과정→국비지원→일정→신청까지 직접 사용하며 이탈 지점을 기록했다.

## 2. 문제 정의

### 2.1 분석 대상 (AS-IS 개요)

빌더 기반 사이트로 학원 소개·과정 안내·국비지원·개강일정·훈련사진·상담·수강신청 메뉴를 갖추고 있었다. 즉 **"정보가 없는 사이트"가 아니라, 정보와 기능은 있으나 신청 판단 흐름과 운영 효율에 맞게 정리되지 않은 사이트**였다.

### 2.4 영향 분석

| 관점 | 영향 |
| --- | --- |
| 예비 수강생 | 신청 판단 정보(과정·비용·일정·절차)가 분산·이미지화 → 의사결정 중 이탈. 구식 인상으로 신뢰 미형성, 신청 후 응답 불투명 |
| 운영자 | 일정·비용 수정마다 이미지 재제작, 신청·문의는 게시판/이메일 수기 확인, 신청서는 종이 스캔 보관 → 반복 업무에 시간이 묶임 |
| 비즈니스 | 신뢰·정보·동선 부재가 곧 상담·신청 전환 손실. 운영 비효율은 인건비·기회비용으로 누적 |

> 국비지원 교육기관에서 **신뢰는 신청 전환의 전제 조건**이므로, 첫인상 신뢰 실패(원인 D)는 미관 문제가 아니라 전환 손실로 직결된다.

### 2.5 핵심 문제 정의

> 예비 수강생은 "내가 들을 수 있는지 · 언제 시작하는지 · 비용은 얼마인지 · 어떻게 신청하는지"를 한 흐름으로 확인하고 싶다. 그러나 기존 사이트는 이 정보가 **이미지·게시글로 흩어져 있고(A), 우선순위가 잘못 잡혀 있으며(D), 신청까지의 동선·피드백이 설계되지 않아(B) 부적합한 게시판으로 처리(C)되어**, 사용자는 판단·실행에서 이탈하고 운영자는 반복 수작업에 묶인다.

## 3. 사용자 정의

> 성격이 정반대인 두 사용자가 있다. **전환시켜야 할 외부 사용자(예비 수강생)와** **운영 부담을 줄여야 할 내부 사용자(운영자)**.

### 3.1 사용자 구조

목적(재취업/창업·부업/내 집 수리/자격증)으로 페르소나를 쪼개면 디자인이 분산되고, 막히는 지점(§2.2의 20개 문제)은 동일하다. 따라서 **주 사용자 1명에 최적화하되 그 외를 배제하지 않고, 목적은 "진입 동기 차이"로 둔다.**

| 구분 | 누구 | 처리 방식 |
| --- | --- | --- |
| Primary | 40~50대 중심, 재취업·이직·구직자 | 디자인·동선 최적화의 기준 |
| Secondary | 창업·부업, 자격증, 청년 구직, 내 집 수리 | 목적별 진입점으로 분기, 별도 페르소나로 분리 안 함 |
| Internal | 운영자(비개발자) | 운영 효율 설계의 대상 |

### 3.2 예비 수강생 (Primary)

| 항목 | 내용 |
| --- | --- |
| 누구 | 40~50대 중심 재취업·이직·구직자 (주력 사용자 설명용 연령) |
| 동기 | 안정적 기술 직무 전환, 국비지원으로 비용 최소화 |
| 디지털 맥락 | 스마트폰 탐색 비중 높음, 전화 상담 신뢰·선호, 복잡한 가입보다 빠른 문의 선호 |
| **핵심 니즈** | **"이 과정을 들으면 진짜 목공을 해서 취업할 수 있나?"** — 실효성·취업 가능성에 대한 확신 |
| 그다음 니즈 | 자비부담금(비용) · 모집/개강 시기 · 신청 방법 |
| 막히는 지점 | 과정·비용이 이미지(4·5·6) / 모집 여부 불명확(7) / 신뢰 미형성(12·14) / 신청 후 응답 불투명(9·10) |

### 3.3 핵심 설계 과제 — 취업률 없이 확신을 만든다

예비 수강생의 1순위 니즈는 "취업 가능성"이지만, **기관은 취업률을 집계하지 않아 화면에 제시할 수 없다.** 가장 강력한 증거가 없는 상황이다.

> **취업을 "보장"하지 않는다. 대신 사용자가 스스로 "할 수 있겠다"고 판단할 "근거"를 충분히 제공한다. 없는 수치를 만들지 않고, 보유 자산으로 확신을 구성한다.**

| 신뢰 신호 (보유) | 답하는 사용자 질문 | 현재 → 방향 |
| --- | --- | --- |
| 회차별 커리큘럼 | "진짜 배우나?" | 이미지(6) → 데이터화 |
| 자격증 경로(건축목공·도장 기능사) | "검증 가능한 결과물 있나?" | 묻힘 → 전면화 |
| 현장 증거(훈련사진·수강일지) | "실제로 이렇게 배우나?" | 작게(8) → 크게·맥락 있게 |
| 수강생 인터뷰·뉴스 영상 | "나 같은 사람도 했나?" | 맥락 보강 |
| 과정 → 직무 연결 | "배우면 무슨 일을 하나?" | 거의 없음 → 신설 |
| 기관 인증(우수훈련기관·이수자평가 A등급) | "믿을 기관인가?" | 보유 (취업 보장으로 과장 안 함) |

> 사용자의 1순위 니즈(취업 가능성)와 보유 자산(취업률 없음)의 갭을 **보장이 아니라 근거 제공으로 메운다**는 것이 이 프로젝트의 핵심 설계 판단이다.

### 3.4 운영자 (Internal)

| 항목 | 내용 |
| --- | --- |
| 누구 | 소규모 운영자(비개발자), 실제 이해관계자 인터뷰 대상 |
| 핵심 니즈 | CMS로 직접 수정 · 신청자 상태를 한눈에 · **웹사이트가 실제 업무에서 편리해야 함** |
| 결정적 맥락 | 수강신청서를 **수기(종이)로 받아 사이트가 사실상 무의미**했음. 사이트가 장식이 아니라 **업무 도구**가 되어야 함 |
| 인터뷰 인용 | *"과정 하나 바꾸려고 이미지를 다시 만든다."* |

운영자의 목표는 "콘텐츠 수정 편의"를 넘어 **"웹사이트를 실제 운영 워크플로우에 연결한다"로** 정의한다. 신청 → 종이 → 스캔 → 수기 보관 대신, 신청 → 데이터 → 상태 관리(접수→상담→등록)로 잇는다.

### 3.5 사용자 정의가 범위를 정당화한다

> 주 사용자의 핵심 행동은 반복 로그인이 아니라 1회성 상담/신청이며 디지털 장벽에 민감하다 → **회원 기능은 전환을 돕는 게 아니라 가로막는 장벽** → 회원가입/로그인/마이페이지를 제외하고 전화 CTA·간편 신청에 집중한다.

"회원 기능을 왜 뺐는가"라는 질문에 직감이 아니라 사용자 정의로 답할 수 있다.

## 4. 목표 & 설계 원칙

### 4.1 목표

| 목표 | 어떤 문제를 푸는가 | 성공의 모습 |
| --- | --- | --- |
| **① 취업 가능성에 대한 확신을 만들어 신청으로 잇는다** | "진짜 취업되나"를 확신 못 해 신청까지 도달 못 함 (12·14, §3.3) | 과정 실효성 납득 → 상담/신청까지 끊김 없이 도달 |
| **② 웹사이트를 실제 운영 업무 도구로 만든다** | 신청서를 수기로 받아 사이트가 무의미 (11, §3.4) | 신청 → 데이터 → 상태관리(접수·상담·등록) |

> 목표 ①을 "전환"이라는 일반어 대신 **"취업 가능성에 대한 확신"으로** 못박는다. 이 프로젝트의 사용자 니즈가 그것이기 때문이다.

| 세부 | 과정·일정·신청·문의·게시글을 분리된 데이터로 관리 | A |
| 세부 | 탐색→판단→신청 전환 흐름과 신청 후 피드백을 설계 | B |
| 세부 | 신청·문의·일정을 목적에 맞는 전용 기능으로 분리(게시판 탈피) | C |
| 세부 | 핵심(과정·정체성·신뢰)을 우선 노출하는 정보 위계 재설계 | D |

> 메타 원칙 ① "콘텐츠를 데이터로"가 이 프로젝트의 대표 메시지다. 사용자 문제와 운영 문제를 동시에 푸는 뿌리이기 때문이다.

### 4.3 측정 지표

실데이터가 없으므로 아래는 성과가 아니라 **"런칭 후 측정하려는 지표(측정 설계)"다**.

- **목표 ① (취업 가능성 → 전환)**
  - 과정 상세 진입률 · 수강신청/상담 폼 제출 수 · 전화 CTA 클릭 수 · 모바일 신청 전환율
- **목표 ② (운영 도구화)**
  - 운영자 콘텐츠 수정 소요 시간(전/후) · 신청 데이터화 여부(수기 → 상태관리)
  - **측정 불가로 제외**
    - 취업률(집계되지 않음. 화면·지표 어디에도 사용하지 않음)

## 5. 범위 · 의사결정

이 절은 "무엇을 만들었나"가 아니라 **"무엇을, 왜 만들지 않았나"를** 다룬다. 모든 범위 결정은 앞 절의 사용자·목표 정의에서 도출됐고, 집계되지 않는 수치나 외부 시스템의 결과를 과장하지 않는 선에서 그어졌다.

### 5.1 핵심 의사결정

| 결정 | 선택 | 근거 / 트레이드오프 |
| --- | --- | --- |
| **회원제 미도입** | 도입 안 함 | 주 사용자 40~50대·1회성·디지털 장벽 민감 → 회원은 장벽. 신청자 관리는 관리자 측 상태관리로 해결(§3.5) |
| **Type A 온라인 폼 미구현** | 이메일 접수 유지 | 신청서 자필 필수 + 연 1회 1회성 → 디지털화 가치 낮음. 범위 절제 |
| **모집상태 라이브 미표시** | 모집중 여부만 수동 | 정원 카운트 신뢰성 확보 어려움 → 의도된 MVP. 잔여석 표시는 2차 |
| **취업률 미표시** | 근거로 대체 | 없는 수치를 만들지 않음(신뢰·윤리). 보장이 아니라 판단 근거 제공(§3.3) |
| **B·C 외부 연동 2차** | 웹은 "안내+가등록"까지 | 결정적 단계가 외부 정부 시스템(고용24·큐넷). "신청 완료"를 흉내내지 않음(사용자 오도 방지) |

### 5.2 기본값을 의심한 결정

| 흔한 기본값 | 따르지 않은 이유 |
| --- | --- |
| 회원제 | 1회성 행동 사용자에게 로그인은 전환 장벽 |
| 온라인 결제 | 수강료는 고용24·현장 상담으로 처리, 불필요 |
| 실시간 채팅 | 전화를 선호하는 사용자층, MVP는 전화·문의폼으로 충분 |

> "없어서 안 넣은 것"이 아니라 **"의심하고 걷어낸 것"이다**. 기본값을 점검한 판단 자체가 이 프로젝트의 성격을 보여준다.
