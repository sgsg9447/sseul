# 유저플로우 — 국비지원 목공 직업훈련기관 웹사이트 리뉴얼

| 항목 | 내용 |
| --- | --- |
| 문서 | 유저플로우 (User Flow) |
| 프로젝트 | 국비지원 목공 직업훈련기관 웹사이트 리뉴얼 |
| 대상 | 실제 운영 중인 직업훈련기관 (사명 비공개) |
| 역할 | 기획 · 설계 · 디자인 · 개발 전 과정 단독 진행 |
| 버전 | v1.0 |

> 예비 수강생의 흐름은 "취업 가능성 확신 → 신청"으로 설계하고, 신뢰가 형성되는 접점(🔵)을 흐름 위에 명시했다. 신청은 과정의 **지원 유형(A·B·C)에서 자동 분기**하며, 접수된 신청은 운영자 워크플로우(접수→상담→등록확인)로 끊김 없이 이어진다.

---

## 1. Flow 1 — 예비 수강생: 탐색 → 판단 → 신청

```
메인 진입
  🔵 인증·이수자평가 A등급·"수료 후 연계(자격·취업·창업)" 노출      ← 첫 신뢰
  ▼
모집중 과정 확인 / 과정 안내 목록 (데이터 카드: 유형·회차·태그)
  ▼
과정 상세
  🔵 NCS 회차별 커리큘럼(구체성) + 자격증 경로                    ← "막연하지 않다"
  🔵 훈련사진·실습 현장(크게)                                    ← "실제로 이렇게 배운다"
  │
  ├─ 비용·자격 불안 → 국비지원 안내(지원율 45~85%, 대상) → 과정 복귀
  └─ 추가 궁금 → 상담문의(바텀시트) / 전화
  ▼
[수강신청하기] → 수강신청 페이지
  ① 모집안내 (신청자격·모집기간·교육일정·진행순서)
                                        fundingType 분기 ↓
  ├─ A 경기도무료 → ② 가등록/문의 → ③ 접수완료 → 서식 다운 + 이메일/방문 접수 → 면접 → 선발
  ├─ B 국비     → ② 가등록/문의 → ③ 접수완료 → 전화 등록확인 → 고용24 수강신청·결제 → 등록완료
  └─ C 자부담   → ② 가등록/문의 → ③ 접수완료 → 전화 등록확인 → 큐넷 시험접수
```

![Flow 1 — 예비 수강생 유저플로우](student-user-flow-notion.png)

> **설계 의도:** "취업 가능성 확신"은 주로 **과정 상세(커리큘럼·자격증)와 훈련사진**에서 형성된다. 그래서 신청 CTA를 그 확신 직후에 배치했다 — 문제 1(행동 위계 부재)의 해결.

---

## 2. 신뢰 신호 접점 (취업 가능성 확신이 형성되는 곳)

| 접점 | 답하는 질문 | 신호 |
| --- | --- | --- |
| 메인 | "믿을 기관인가?" | 인증·A등급·수료 후 연계 |
| 과정 상세 | "진짜 배우나? 결과물 있나?" | NCS 커리큘럼·자격증 경로 |
| 훈련 사진 | "실제로 이렇게 배우나?" | 실습 현장 그리드 |
| 국비지원 | "비용은? 받을 수 있나?" | 지원율·지원 대상 |

> 확신은 단일 화면이 아니라 **접점들의 누적**으로 만들어진다(취업률 없이 근거로 확신을 구성).

---

## 3. Flow 2 — 운영자 워크플로우

```
사이트 신청/문의 접수 (→ 데이터로 저장)
  ▼
대시보드: 신규 신청·문의·모집중 과정 한눈에
  ▼
신청 관리: 접수 → 전화상담 → (유형별 분기)
        A: 면접 → 선발 → 훈련참여
        B·C: 외부(고용24/큐넷) 진행 안내 → 등록확인(수동)
  ▼
문의 관리: 답변대기 → 답변완료 (Q&A)
```

![Flow 2 — 운영자 워크플로우](admin-user-flow-notion.png)

> **B·C의 한계와 처리:** 고용24·큐넷의 진행 결과를 사이트가 자동으로 알 수 없다. 따라서 운영자가 전화·고용24 조회로 확인한 뒤 상태를 **수동으로 '등록확인'으로 전이**한다. 외부 연동은 2차 과제로 둔다(과장하지 않음).

핵심: 이전엔 이메일·종이로 흩어졌던 접수가, 이제 **접수부터 상태관리까지 한 흐름**으로 이어진다 — 목표②(사이트=업무 도구)가 플로우로 증명되는 지점.

---

## 4. 이 플로우가 푸는 문제

| 흐름상 지점 | 해결한 문제 |
| --- | --- |
| 메인→과정상세 신뢰 접점 | 문제 12·14 (신뢰 형성) |
| 과정 상세 = 데이터 | 문제 5·6 |
| 신청 CTA를 확신 직후 배치 | 문제 1 (행동 위계) |
| 지원 유형별 신청 분기 | 과정마다 제각각이던 신청을 일관되게 |
| 접수 → 상태관리(수동 등록확인 포함) | 문제 10·11 (신청 후 깜깜·수기 보관) |
| 상담문의 답변대기→완료 | 문제 9 (피드백 부재) |

## 5. 플로우에서 드러난 의사결정 (→ 서비스 기획서 §5)

| 결정 | 한 줄 |
| --- | --- |
| 회원 기능 제외 | 1회성 상담/신청 사용자에게 로그인은 장벽 |
| Type A 온라인 폼 미구현 | 자필 필수·연 1회 → 이메일 접수 유지 |
| B·C 외부 연동 2차 | 결정적 단계가 외부 정부 시스템 |
| 모집상태 라이브 미표시 | 운영 단순 ↔ 자가판단의 트레이드오프(2차) |
